# Кабинет юриста — мини-CRM (тестовое задание)

Живая ссылка: **https://takemyenergy13.github.io/lawyer-crm/**

Одностраничный дашборд для юриста: таблица клиентов, добавление клиента (имя, телефон, статус дела), смена статуса прямо в строке, живые счётчики по статусам с фильтрацией. Данные общие для всех посетителей и обновляются в реальном времени (открой в двух вкладках — изменения синхронизируются без перезагрузки).

## Стек и почему

- **React + Vite + TypeScript** — быстрый старт, типобезопасность, знакомый стек.
- **Firebase Firestore** (вариант «React + Firebase» из ТЗ) — реальная база с realtime-подпиской `onSnapshot`, работает напрямую из статики без своего бэкенда.
- **GitHub Pages + GitHub Actions** — деплой на каждый push в `main`, тесты гоняются в CI перед сборкой.
- **Telegram-уведомление** — при добавлении клиента фронт шлёт fire-and-forget запрос на n8n-webhook, n8n доставляет сообщение в Telegram. Токен бота живёт на сервере n8n, не в клиенте; недоступность вебхука не ломает добавление клиента.

## Архитектура

```
Браузер (React SPA) ──realtime──▶ Firestore (коллекция clients)
        │
        └──POST при добавлении──▶ n8n webhook ──▶ Telegram Bot API
```

Вся производная логика (счётчики, фильтр) — чистые функции (`src/lib/clients-logic.ts`), покрыты unit-тестами (vitest).

## Осознанные компромиссы (тестовое ≠ прод)

- **Firestore rules открыты** (test mode): любой посетитель читает и пишет коллекцию `clients`. Для прода — Firebase Auth + rules по владельцу.
- **Firebase web config в коде** — это публичные идентификаторы, так задумано Firebase; секретов в клиенте нет.
- **n8n webhook URL виден в бандле** — худший сценарий: кто-то дёрнет вебхук и пришлёт лишнее уведомление. Токен бота при этом не раскрывается.
- Нет редактирования/удаления клиентов, авторизации, пагинации — вне скоупа 2–3 часов.

## Запуск локально

```bash
npm install
npm run dev      # http://localhost:5173/lawyer-crm/
npm test         # unit-тесты логики
```
