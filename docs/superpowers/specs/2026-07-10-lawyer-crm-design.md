# Lawyer CRM — тестовое задание (design doc)

Дата: 2026-07-10. Бюджет: ~2–2.5 часа чистой работы. Дедлайн сдачи: 24 часа с отклика.

## Цель

Работающий прототип мини-CRM для юриста: таблица клиентов, добавление клиента (имя, телефон, статус дела), смена статуса, счётчики по статусам. Живая ссылка + репозиторий + краткий лог. Бонус: Telegram-уведомление при добавлении клиента.

## Стек и деплой

- **Frontend:** Vite + React + TypeScript, один SPA-экран (дашборд).
- **БД:** Firebase Firestore, клиентский web SDK (работает со статики). Firebase web config в бандле — публичный by design.
- **Деплой:** GitHub Pages через GitHub Actions (push в `main` → build → deploy). `base` в vite.config под имя репо.
- **Уведомления:** личный n8n пользователя — фронт делает fire-and-forget `fetch` POST на webhook, n8n-workflow шлёт сообщение через Telegram-ноду. Токен TG-бота хранится в n8n, не в клиенте.

## Модель данных

Коллекция `clients`:

```ts
{
  name: string;        // required
  phone: string;       // freeform, без маски
  status: 'new' | 'in_progress' | 'closed';
  createdAt: Timestamp; // serverTimestamp()
}
```

Firestore security rules: открытые read/write только на коллекцию `clients`. Осознанный компромисс для тестового, фиксируется в README.

## UI (один экран)

1. Шапка с названием продукта.
2. Три счётчика-карточки: «Новый» / «В работе» / «Закрыт». Живые (считаются из snapshot), клик по карточке фильтрует таблицу (повторный клик — сброс фильтра).
3. Таблица клиентов: имя, телефон, статус-бейдж, дата добавления. Статус меняется прямо в строке — dropdown на бейдже, любой переход разрешён.
4. Форма добавления клиента: имя (required) + телефон + статус (default «Новый»).
5. Realtime: `onSnapshot` на коллекцию — таблица и счётчики обновляются без перезагрузки у всех открытых вкладок.
6. Empty state для пустой таблицы и для пустого результата фильтра.
7. Стиль: аккуратный дашборд (скилл design-engineering), без тяжёлой арт-дирекции. Базовая отзывчивость, десктоп-first.

## Поток данных

- Чтение: `onSnapshot(collection('clients'), orderBy('createdAt','desc'))` → state → таблица + счётчики (reduce по status).
- Добавление: `addDoc` → параллельно `fetch(N8N_WEBHOOK_URL, {method:'POST', body:{name, phone}})` без await результата; ошибка вебхука глотается (console.warn), UX не блокируется.
- Смена статуса: `updateDoc(doc, {status})`.

## Обработка ошибок

- Ошибка Firestore при добавлении/обновлении → inline-сообщение об ошибке, данные формы не теряются.
- Недоступный n8n → уведомление молча не уходит, ядро работает.
- Невалидная форма (пустое имя) → блокировка submit с подсказкой.

## Telegram-бонус (n8n)

Workflow на личном n8n: Webhook (POST, respond immediately) → Telegram sendMessage «🆕 Новый клиент: {name}, {phone}». URL вебхука и доступ к n8n даёт пользователь на шаге интеграции.

## Вне скоупа

Авторизация, редактирование и удаление клиентов, пагинация, мобильная полировка сверх базовой, тёмная тема.

## Порядок работы

1. Пользователь: Firebase-проект + web app (даёт config), TG-бот через BotFather, доступ к личному n8n. → check: config/доступы получены.
2. Скаффолд, Firestore-слой, UI. → check: локально работает с живой базой.
3. n8n workflow для уведомления. → check: тестовое сообщение пришло в TG.
4. GitHub repo, Actions, Pages. → check: живая ссылка, смоук через gstack (рендер, консоль, добавление клиента).
5. README + краткий лог сдачи (что сам / что AI, почему стек, время начала и конца).
